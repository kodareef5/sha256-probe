# 1. Why FACE fails in practice

Because you built it as an **interpreter**, not a **compiler**.

Your current FACE cost model is:

- branch,
- run generic GF(2) reductions over a large variable set,
- repeat per bit/per branch.

That is asymptotically fine, but at N≤16 the constant is fatal.

## What must change

### A. Stop RREF-ing all 4N variables per branch
Maintain linear constraints only on the **current frontier variables**, not the whole 128-bit message vector.

At a chunk boundary, most variables are dead:
- already assigned,
- or no longer referenced by any future rotation/schedule term.

Project them out immediately.

### B. Stop canonicalizing at every bit
Do cheap incremental propagation inside a chunk.
Do full canonical RREF only at **chunk boundaries**.

This alone cuts the symbolic overhead by a large constant.

### C. Stop copying contexts
Use a rollback trail.

```c
struct LinCtx {
    Row rows[MAXR];
    uint8_t pivot_of_var[MAXV];
    uint16_t nrows;
    Trail trail[TRAIL_MAX];
    uint32_t ttop;
};
```

Every `add_constraint()` pushes row/pivot changes onto `trail`.
On backtrack, restore by popping.
No memcpy of 2–8 KB per branch.

### D. Stop branching on raw bit values inside nonlinear ops
Branch on **modes**.

This is the biggest one.

For one binary adder bit `x + y + cin -> z, cout`, do **3-way mode branching**:

- `MODE_00`: `x=0, y=0`
- `MODE_01`: `x xor y = 1`
- `MODE_11`: `x=1, y=1`

Then:
- `z` is affine,
- `cout` is determined,
- the constraints on `x,y` are linear.

For `Ch` and `Maj`:

- `Ch(e,f,g)`: branch on `q = e`
  - `q=0 => Ch = g`
  - `q=1 => Ch = f`

- `Maj(a,b,c)`: branch on `p = b xor c`
  - `p=0 => Maj = c`
  - `p=1 => Maj = a`

So the whole chunk becomes:
- small branching on nonlinear control bits,
- everything else linear.

That is much cheaper than branching on raw operand values.

### E. Stop using FACE online
Use FACE once to **compile a chunk transducer**.
Then run DP over compiled state IDs.

That is the real answer to Q1.

## Within a chunk

Process bit-by-bit *inside* the chunk.

At each bit:
1. fetch the affine forms for needed register bits,
2. form `Sigma` terms by XOR of forms,
3. branch on `Ch`/`Maj` selectors,
4. branch on adder modes,
5. add linear constraints,
6. propagate,
7. continue.

At chunk end:
1. eliminate chunk-internal variables,
2. keep only future frontier vars,
3. RREF,
4. intern to `lin_id`,
5. pack outgoing carries,
6. emit successor `StateKey`.

# 5. How to make `lin_id` practical

This is where FACE becomes usable.

## Use two contexts

### Inside chunk: mutable `LinCtx`
Cheap incremental operations with rollback.

### At chunk boundary: canonical `CanonCtx`
Project to frontier, row-reduce, normalize row order, hash-cons.

## Frontier projection
At chunk boundary, only keep variables that can still influence future chunks.

For N=32, B=8, a safe first frontier is:
- all unassigned bits of `W1[57..60]` in later chunks,
- plus any same-word future bits referenced by rotations into earlier processed chunks.

Yes, that’s overinclusive.
That’s okay for v1.

## Canonicalization
After eliminating internal/dead vars:

1. renumber surviving vars by a fixed global order,
2. RREF,
3. remove zero rows,
4. serialize rows+rhs,
5. hash-cons to a unique table,
6. return `lin_id`.

Equivalent suffix problems collapse to the same `lin_id`.
That is where the exponential gain comes from.
