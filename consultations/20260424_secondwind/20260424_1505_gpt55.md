## My highest-EV pick

> **Wang-style block-2 modification at N=32.**

Reason: it is the only open frontier that is structurally outside the closed single-block cascade box.

The previous “naive multi-block” result is not enough to close it. What was closed was:

> “Can a naive second block cancel generic/high-HW residuals easily?”

What remains open is:

> “Can we design a Wang-style differential path for block 2 tailored to the residual family produced by a cascade near-collision?”

That is a different mechanism.

## What exactly is missing

You do **not** need more random block-2 SAT first.

You need a **block-2 bitcondition/trail engine**:

1. Input: IV difference class from block-1 near-collision residuals.
2. Output: differential trail over block 2 with message modification opportunities.
3. Search objective:
   - maximize satisfied local conditions through rounds 0–31,
   - not solve full 64 immediately.
4. Use literature-style steps:
   - disturbance correction,
   - neutral bits,
   - local collisions,
   - message modification,
   - SAT only after trail exists.

## First concrete block-2 sprint

1. Collect 100–1000 best block-1 near-collision residuals, not just one.
2. Cluster residuals by:
   - active registers,
   - bit positions,
   - signed/additive vs XOR form,
   - carry-entry profile.
3. For each residual cluster, run a reduced-round block-2 search:
   - 20, 24, 28, 32 rounds,
   - full W[0..15] message freedom,
   - enforce a low-HW differential trail rather than final equality.
4. Compare against known SHA-256 differential trail frameworks.

Headline path:
- “2-block cascade+Wang construction reaches X full-schedule rounds”
- or “block-2 absorbs cascade residual through Y rounds with full schedule.”

Even a strong negative here is informative, because it closes the only structurally new frontier.

## My critique of this pick

It is human-time expensive. You will need to read and implement classical SHA-2 message modification, not just launch solvers.

If you are unwilling to build a real bitcondition/trail engine, then block-2 should not be started. Naive SAT on block 2 will likely reproduce the existing 18-round frontier and stall.

# 5. Living-list schema

## Format choice

Use:

- **YAML** for canonical curated registries.
- **JSONL** for append-only run/event logs.
- **Generated Markdown** for human-readable tables.

Reason:
- Markdown tables are bad canonical data.
- JSON is noisy for humans.
- YAML is readable and diffable.
- JSONL is ideal for run events.

## Global maintenance rules

1. Every registry item has:
   - `id`
   - `status`
   - `owner`
   - `last_updated`
   - `evidence_level`
   - `next_action`
   - `kill_or_close_criteria`
2. Status enum:
   - `open`
   - `in_flight`
   - `blocked`
   - `closed`
   - `reopened`
   - `archived`
3. Every compute run appends one row to `registry/runs.jsonl`.
4. Any SAT/UNSAT/CNF result must include:
   - encoder commit,
   - CNF hash,
   - sr-level audit result,
   - solver version,
   - seed,
   - wall time,
   - CPU time if available.
5. Staleness:
   - `in_flight` stale after 7 days without update.
   - `open` stale after 30 days.
   - stale items require either update, close, or explicit defer.
6. Closed items require:
   - close reason,
   - evidence files,
   - “would change my mind” trigger.
